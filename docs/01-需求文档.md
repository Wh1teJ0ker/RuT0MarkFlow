# Markdown 编辑器需求文档

## 1. 文档定位
- 本文档定义当前版本的产品需求边界。
- 当前文档中的需求默认即本项目的第一期 MVP 范围。
- MVP 的目标不是一次性做成终态产品，而是先完成一个可稳定使用的本地 Markdown 工作区编辑闭环。
- MVP 优先保证主流程完整可用：选择工作区、建立索引、打开文档、编辑保存、渲染预览、模式切换、基础状态提示、长文档可用性。
- 后续扩展能力应在 MVP 验收通过后再进入下一阶段。

## 2. 项目目标
- 开发一个基于 Rust 和 Tauri 的桌面 Markdown 编辑器。
- 面向本地 Markdown 文档的写作、整理、阅读和维护场景。
- 提供以文件夹为驱动的工作区体验，而不是以单文件临时打开为核心。
- 提供兼顾阅读体验与编辑效率的双模式使用方式。
- 在常规文档和超长文档场景下，都保持稳定、流畅、可恢复的使用体验。

## 3. MVP 范围

### 3.1 本期必须实现
- 选择本地文件夹作为工作区。
- 自动扫描工作区内的 Markdown 文件并建立索引。
- 在左侧展示工作区 Markdown 索引，并支持层级目录展示。
- 支持点击索引项打开对应 Markdown 文档。
- 支持新建、编辑、保存、另存为 Markdown 文档。
- 支持无感 Markdown 渲染模式。
- 支持双栏编辑模式。
- 支持常见 Markdown 语法渲染。
- 支持行内公式和块级公式渲染。
- 支持相对路径图片、相对路径链接、工作区内 Markdown 链接解析。
- 支持未保存状态提示、文件读写失败提示、索引失败提示、资源加载失败提示。
- 支持最近工作区、最近文档、最近模式的基础状态恢复。
- 支持长文档和超长 Markdown 文档场景下的基础性能优化。

### 3.2 本期明确不做
- 云同步。
- 多人协同编辑。
- 账号系统。
- 在线发布。
- 插件系统。
- AI 写作、自动总结、自动改写等智能能力。
- 文档评论、批注、审阅流。
- 跨设备同步。
- 完整所见即所得富文本体系。
- 面向公网内容管理的能力。

### 3.3 MVP 完成标准
- 主流程可从空启动走到实际编辑和保存，不依赖人工绕行。
- 关键状态可被用户感知，包括加载中、未保存、保存成功、保存失败、索引更新中。
- 超长文档场景下不要求做到极致性能，但不能出现频繁卡死或无法继续使用的情况。
- 对复杂边缘场景允许做降级处理，但不能因为单个文档或单个资源异常导致整个应用不可用。

## 4. 用户与使用场景

### 4.1 目标用户
- 习惯使用本地 Markdown 文件进行知识整理的个人用户。
- 需要维护一个文档文件夹或笔记仓库的用户。
- 希望在桌面应用中兼顾源码编辑与阅读体验的用户。

### 4.2 核心使用场景
- 打开本地笔记目录，浏览其中的 Markdown 文件并快速切换阅读。
- 编辑某篇 Markdown 文档，并实时查看渲染效果。
- 在工作区中通过索引或相对链接在不同 Markdown 文档之间跳转。
- 阅读包含图片、表格、代码块、公式的技术文档。
- 处理篇幅较长、结构复杂的 Markdown 文档，而不会因为渲染导致明显卡顿。

## 5. 核心功能需求

### 5.1 工作区需求
- 支持用户选择一个本地文件夹作为工作区。
- 应用的核心入口应为工作区，而不是单文件打开。
- 选择工作区后，系统应自动扫描其中的 Markdown 文件。
- 支持识别工作区子目录中的 Markdown 文件。
- 工作区切换后，页面内容、索引内容和当前上下文应同步切换。
- 工作区不可访问时，应给出明确提示并允许用户重新选择。

### 5.2 索引需求
- 支持为当前工作区建立 Markdown 文件索引。
- 索引中至少应包含文件名、相对路径、所属目录、文件类型和基础排序依据。
- 索引应支持按目录层级展示。
- 支持点击索引项直接打开对应 Markdown 文档。
- 当前打开文档在索引中应有明确高亮。
- 工作区内容变化后，索引应可更新。
- 索引建立失败或更新失败时，应给出可理解提示。
- 索引的存在应帮助提升文件列表展示和文档切换效率。

### 5.3 文档操作需求
- 支持新建 Markdown 文档。
- 支持打开 Markdown 文档。
- 支持编辑 Markdown 内容。
- 支持保存当前文档。
- 支持文档另存为。
- 支持关闭当前文档或切换到其他文档。
- 当前文档被修改后，应显示未保存状态。
- 当前文档未保存时，切换文档、切换工作区或关闭窗口前应提示用户。

### 5.4 使用模式需求
- 支持双模式使用。
- 支持无感 Markdown 渲染模式。
- 支持双栏编辑模式。
- 无感 Markdown 渲染模式下，用户看到的主要应是格式化后的内容结构，尽量接近阅读与轻编辑体验。
- 双栏编辑模式下，左侧为源码编辑区，右侧为渲染预览区。
- 支持在两种模式之间直接切换。
- 模式切换后当前文档内容、当前滚动上下文、保存状态和基础编辑上下文不应丢失。
- 两种模式下的渲染结果应尽量一致。

### 5.5 编辑器需求
- 支持纯文本输入。
- 支持文本删除与修改。
- 支持撤销与重做。
- 支持剪切、复制、粘贴。
- 支持常用快捷键。
- 支持显示当前文档名称。
- 支持显示当前文档是否已修改。
- 支持空文档状态和新建文档状态。

## 6. Markdown 渲染需求

### 6.1 MVP 必须支持的语法范围
- 标题，包括一到六级标题。
- 普通段落。
- 换行与空行。
- 粗体、斜体、粗斜体、删除线。
- 行内代码。
- 无序列表、有序列表、任务列表、嵌套列表。
- 引用块、嵌套引用块。
- 分隔线。
- 围栏代码块。
- 带语言标记的代码块。
- 表格、表头、对齐方式。
- 行内链接。
- 自动识别可点击链接。
- 图片。
- 行内公式。
- 块级公式。

### 6.2 渲染行为要求
- 支持 Markdown 实时渲染。
- 支持在无感 Markdown 渲染模式下展示格式化内容。
- 支持在双栏编辑模式下展示预览内容。
- 文档内容变化后，渲染结果应及时刷新。
- 渲染刷新不应明显打断用户输入。
- 渲染结果应尽量还原 Markdown 文档的结构层级和阅读体验。
- 对不支持的扩展语法，应保证不导致编辑器异常。
- 对无法正确渲染的语法片段，应尽量以可阅读方式降级展示。
- 支持浅色和深色主题下的基础可读性。

### 6.3 图片、链接与资源处理
- 支持渲染本地图片资源。
- 支持渲染相对路径图片资源。
- 图片加载失败时应给出可感知反馈。
- 支持在工作区内解析相对路径链接。
- 点击文档中的工作区内 Markdown 相对链接时，应支持打开对应 Markdown 文档或资源。
- 对外部链接应明确区分应用内跳转和系统外部打开行为。

### 6.4 公式渲染要求
- 支持公式渲染。
- 支持行内公式渲染。
- 支持块级公式渲染。
- 支持常见 LaTeX 数学公式写法。
- 公式渲染结果应具备可读性。
- 公式渲染失败时，应尽量保留原始公式内容或提供明确反馈。

### 6.5 本期不作为验收必选项的扩展语法
- 高亮文本语法。
- 脚注语法。
- 自动目录块。
- 锚点导航增强能力。
- 自定义容器语法。

说明：上述能力后续可以扩展支持，但不作为本期 MVP 的必须交付项。

## 7. 界面与交互需求
- 提供工作区选择入口。
- 提供左侧工作区索引区。
- 提供中央内容区。
- 提供模式切换入口。
- 提供新建、保存、另存为入口。
- 提供状态栏或等价的状态展示区域。
- 支持显示当前工作区路径或名称。
- 支持显示当前文件名。
- 支持显示当前保存状态。
- 启动应用后，在没有工作区时应展示空状态提示。
- 选择工作区后，应展示加载状态并在索引可用后进入正常工作区界面。
- 当前文档未保存时，切换文档、切换工作区或关闭窗口需要提示用户。

## 8. 文件与状态需求

### 8.1 文件类型
- 支持 `.md` 文件。
- 支持 `.markdown` 文件。
- 是否支持 `.txt` 作为普通文本打开，不作为 MVP 必做项。

### 8.2 状态管理
- 支持记录最近一次打开的工作区。
- 支持记录最近一次打开的文档。
- 支持下次启动时恢复上次工作区。
- 支持下次启动时恢复上次打开文档。
- 支持保留当前使用模式。
- 状态恢复失败时，不应阻塞应用进入可用状态。

### 8.3 主题需求
- 支持浅色主题。
- 支持深色主题。
- 主题切换不应破坏当前文档状态和模式状态。

### 8.4 图标规范
- 界面中的图标必须来自统一的矢量图标库，不允许使用 emoji 表情作为功能图标。
- 图标库应轻量、可按需引入，并兼顾浅色与深色主题。
- 同一语义的操作在不同位置应使用一致图标。

## 9. 非功能需求

### 9.1 性能需求
- 应用启动后应能较快进入可交互状态。
- 工作区加载过程需要可感知，但不应卡死界面。
- 常规 Markdown 文档编辑过程需要保持流畅。
- 文档切换过程需要较快响应。
- 预览刷新需要平滑。
- 基于索引的文件加载和渲染准备应优于每次重新全量扫描。

### 9.2 超长文档性能要求
- 支持超长 Markdown 文档渲染与显示。
- 超长 Markdown 文档在打开、显示、滚动和切换模式时不应出现明显卡顿。
- 需要针对超长 Markdown 文档的渲染与显示性能进行优化。
- 在性能压力较高时，允许采用节流、缓存、分段渲染、虚拟化或降级显示策略。
- 即使部分资源加载失败或公式渲染较慢，也不应导致整个界面长期失去响应。

### 9.3 稳定性与可理解性需求
- 错误提示需要易于理解。
- 单个文件、单张图片或单个公式异常不应导致整个应用崩溃。
- 默认情况下用户内容只保存在本地。
- 应用不应主动上传用户文档内容。

## 10. 平台需求
- 支持 macOS。
- 支持 Windows。
- Linux 可作为后续兼容目标，不作为本期强制验收平台。

## 11. 验收标准

### 11.1 主流程验收
- 用户可以选择一个本地文件夹作为工作区。
- 系统以文件夹为驱动加载工作区内容。
- 系统可以自动扫描并展示工作区中的 Markdown 索引列表。
- 用户可以通过点击 Markdown 索引项直接打开对应文档。
- 用户可以在索引列表中快速打开和切换文档。
- 用户可以正常新建、编辑、保存和另存为 Markdown 文档。
- 未保存文档在关闭、切换文档或切换工作区时会提示用户。

### 11.2 模式与渲染验收
- 系统支持无感 Markdown 渲染模式和双栏编辑模式。
- 两种模式都可以正确显示 Markdown 内容。
- 模式切换后文档内容、保存状态和当前上下文不丢失。
- 系统可以正确渲染本期要求的常见 Markdown 语法。
- 系统支持行内公式和块级公式渲染，并能够展示常见 LaTeX 数学公式。
- 系统可以在工作区内正确解析相对路径图片和相对链接。
- 工作区内 Markdown 链接可以触发应用内打开对应文档。

### 11.3 索引与状态验收
- 系统可以为工作区建立并使用索引。
- 工作区内文件变化后，文件列表与索引可以更新。
- 系统可以显示当前工作区、当前文档、当前模式和当前保存状态。
- 系统可以恢复最近一次工作区、最近文档和最近模式，或在恢复失败时安全降级到空工作区状态。

### 11.4 性能与稳定性验收
- 在常规开发机器上，加载中大型工作区时界面保持可操作，不出现持续性假死。
- 以不少于 1000 个 Markdown 文件的工作区作为压力场景时，索引构建过程应可感知且界面可交互。
- 以不少于 10000 行或约 1 MB 文本量级的 Markdown 文档作为超长文档基线场景时：
  - 打开文档后应能在可接受时间内显示首屏可视内容。
  - 滚动过程中不应出现持续 1 秒以上的明显卡死。
  - 模式切换过程中不应出现持续 1 秒以上的明显卡死。
  - 即使存在大量图片、代码块或公式，也应优先保证可视区域可阅读。
- 文件读取失败、文件保存失败、索引建立失败、图片加载失败、公式渲染失败时，应用应给出明确反馈且主界面仍可继续使用。

## 12. 本文档输出边界
- 本文档回答“做什么、先做什么、不做什么、做到什么程度算完成”。
- 本文档不展开具体第三方库选型与实现代码结构。
- 具体页面布局和操作细节以页面与交互说明为准。
- 具体模块划分、命令接口、数据结构和性能实现方案以技术设计文档为准。
