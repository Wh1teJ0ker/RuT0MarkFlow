# 开发任务清单

## 1. 文档目标
- 将需求文档、页面与交互说明、技术设计文档拆解为可以直接执行的开发任务。
- 为后续实际编码、联调、测试和验收提供顺序化任务清单。
- 当前清单以 MVP 为目标，优先保障主流程闭环，而不是一次性实现所有远期扩展能力。
- 本文档强调任务依赖、产出物、落地建议、验收条件、风险点和验证方式。

## 2. 使用方式说明
- 每个任务都应至少明确：目标、前置依赖、建议修改位置、实现要点、交付物、验收标准、验证方式。
- 开发时建议按阶段推进，但允许在不破坏依赖关系的前提下局部并行。
- 若实现阶段与文档存在偏差，应先同步需求文档或技术设计文档，再继续编码。
- 对 MVP 无直接贡献的优化项，除非明显阻塞主流程，否则不应提前插队。

## 3. MVP 完成判定
满足以下条件，才可视为本期 MVP 基本完成：
- 可以选择工作区并加载 Markdown 索引。
- 可以从索引点击打开文档。
- 可以新建、编辑、保存和另存为文档。
- 可以在无感渲染模式和双栏编辑模式之间切换。
- 可以渲染本期约定的 Markdown 基础语法、图片、链接和公式。
- 可以处理未保存提示、文件读写失败、索引失败、资源加载失败。
- 可以在长文档和超长文档场景下维持可用，不出现频繁卡死。
- 可以完成一次基于需求文档的完整验收。

## 4. 建议阶段划分
建议按以下阶段推进：
- 阶段一：项目初始化与基础骨架
- 阶段二：工作区与索引能力
- 阶段三：文档读写与编辑能力
- 阶段四：Markdown 渲染与公式能力
- 阶段五：模式切换与状态管理
- 阶段六：性能优化与稳定性处理
- 阶段七：测试、验收与收尾

## 5. 任务清单

### 5.1 阶段一：项目初始化与基础骨架

#### T01：初始化 Tauri 项目
- 目标：建立可运行的 Rust + Tauri + 前端工程骨架。
- 前置依赖：无。
- 建议修改位置：
  - `src-tauri/`
  - `src/`
  - 根目录项目配置文件
- 实现要点：
  - 创建 Tauri 应用基础结构。
  - 接入前端构建入口。
  - 确认开发模式、构建模式和桌面启动流程可跑通。
- 交付物：
  - 可启动的桌面窗口。
  - 基础前端页面入口。
  - 基础 Tauri 配置文件。
- 验收标准：
  - 本地可执行开发启动命令。
  - 窗口能正常打开且加载前端页面。
- 验证方式：
  - 手动启动应用。
  - 检查控制台无阻塞性错误。
- 风险点：
  - 前端打包路径与 Tauri 配置不一致。
  - 本地开发环境未安装 Tauri 依赖。

#### T02：建立基础目录结构
- 目标：按职责建立前端与 Rust 模块目录。
- 前置依赖：T01。
- 建议修改位置：
  - `src/components/`
  - `src/modules/`
  - `src/stores/`
  - `src/services/`
  - `src/types/`
  - `src-tauri/src/commands/`
  - `src-tauri/src/modules/`
  - `src-tauri/src/models/`
- 实现要点：
  - 目录命名与职责边界保持一致。
  - 避免把所有逻辑直接堆到入口文件。
- 交付物：
  - 结构化目录树。
- 验收标准：
  - 前端和 Rust 侧核心职责均有对应目录。
- 验证方式：
  - 代码审查目录结构。
- 风险点：
  - 目录过早细分导致实际落地困难。

#### T03：建立基础应用框架
- 目标：搭建主窗口页面骨架和前后端通信入口。
- 前置依赖：T01、T02。
- 建议修改位置：
  - `src/app/App.*`
  - `src/components/layout/`
  - `src/services/tauri/`
  - `src-tauri/src/main.rs`
- 实现要点：
  - 建立顶部工具栏、左侧栏、中央内容区、底部状态栏的页面骨架。
  - 建立统一的 Tauri command 调用封装。
- 交付物：
  - 基础页面框架。
  - command 调用工具层。
- 验收标准：
  - 页面骨架可显示。
  - 前端能成功调用至少一个 Rust command。
- 验证方式：
  - 手动点击触发一个测试命令。
- 风险点：
  - 命令调用层分散，后续难统一错误处理。

#### T04：建立基础类型与错误响应规范
- 目标：统一前后端使用的数据结构和错误结构。
- 前置依赖：T03。
- 建议修改位置：
  - `src/types/`
  - `src-tauri/src/models/`
  - `src-tauri/src/modules/errors/`
- 实现要点：
  - 定义工作区、索引项、文档状态、设置项、错误响应等基础结构。
  - 统一 success/data/error 格式。
- 交付物：
  - 前后端共享的类型定义。
  - Rust 错误返回结构。
- 验收标准：
  - 后续命令均使用统一响应结构。
- 验证方式：
  - 通过一个成功响应和一个失败响应进行联调。
- 风险点：
  - 类型定义频繁漂移，导致前后端字段不一致。

### 5.2 阶段二：工作区与索引能力

#### T05：实现工作区选择能力
- 目标：支持用户选择本地文件夹作为工作区。
- 前置依赖：T03、T04。
- 建议修改位置：
  - `src/components/toolbar/`
  - `src/modules/workspace/`
  - `src/services/tauri/`
  - `src-tauri/src/commands/workspace.rs`
- 实现要点：
  - 接入系统文件夹选择对话框。
  - 选择后校验路径是否存在、是否可访问。
  - 空工作区状态下允许直接进入选择流程。
- 交付物：
  - 选择工作区按钮可用。
  - 工作区路径可从 Rust 返回前端。
- 验收标准：
  - 用户可以通过界面选择一个本地文件夹。
  - 无效路径或取消选择时界面状态正确。
- 验证方式：
  - 手动选择合法目录、取消选择、模拟不可访问目录。
- 风险点：
  - 不同平台对目录权限反馈不一致。

#### T06：实现空工作区与工作区加载状态
- 目标：在页面上明确区分未选择工作区和正在加载工作区。
- 前置依赖：T05。
- 建议修改位置：
  - `src/components/layout/`
  - `src/modules/workspace/`
  - `src/stores/`
- 实现要点：
  - 定义 `idle/loading/ready/error` 等工作区状态。
  - 空状态下显示选择工作区提示。
  - 加载状态下显示轻量进度反馈。
- 交付物：
  - 空工作区页面。
  - 工作区加载中页面状态。
- 验收标准：
  - 用户能明确知道当前是否已选择工作区。
- 验证方式：
  - 手动检查状态切换。
- 风险点：
  - 加载状态与错误状态混淆。

#### T07：实现工作区扫描能力
- 目标：扫描工作区内的 Markdown 文件。
- 前置依赖：T05、T06。
- 建议修改位置：
  - `src-tauri/src/modules/workspace/scanner.rs`
  - `src-tauri/src/modules/workspace/mod.rs`
- 实现要点：
  - 递归扫描工作区。
  - 识别子目录结构。
  - 过滤非 `.md` 和 `.markdown` 文件。
  - 对不可读文件进行跳过并记录问题。
- 交付物：
  - 可复用的扫描函数。
- 验收标准：
  - 能返回工作区下所有目标 Markdown 文件。
- 验证方式：
  - Rust 单元测试。
  - 手动使用含多级目录的样例工作区验证。
- 风险点：
  - 大目录扫描导致启动阻塞。

#### T08：实现索引数据结构
- 目标：定义前后端统一使用的索引对象。
- 前置依赖：T04、T07。
- 建议修改位置：
  - `src/types/`
  - `src-tauri/src/models/workspace.rs`
- 实现要点：
  - 定义扁平索引项结构。
  - 定义目录树节点结构。
  - 明确相对路径、父目录路径、显示名字段。
- 交付物：
  - `IndexEntry`。
  - `IndexTreeNode`。
- 验收标准：
  - 前后端能够用同一套字段渲染索引。
- 验证方式：
  - 示例数据联调。
- 风险点：
  - 只定义树结构会导致前端查找当前文档效率低。

#### T09：实现索引构建流程
- 目标：把扫描结果转换为前端可直接消费的索引结构。
- 前置依赖：T07、T08。
- 建议修改位置：
  - `src-tauri/src/modules/workspace/indexer.rs`
  - `src-tauri/src/commands/workspace.rs`
- 实现要点：
  - 根据扫描结果构建扁平数组和目录树。
  - 生成稳定 ID。
  - 统一排序规则，例如目录优先、名称排序。
- 交付物：
  - 工作区索引构建函数。
  - `load_workspace` 返回结果。
- 验收标准：
  - 前端收到索引后可直接渲染层级结构。
- 验证方式：
  - 手动加载工作区并检查索引展示顺序。
- 风险点：
  - 排序或路径处理不一致导致高亮与打开错位。

#### T10：实现左侧索引区 UI
- 目标：把工作区索引在界面中可视化展示出来。
- 前置依赖：T06、T09。
- 建议修改位置：
  - `src/components/sidebar/`
  - `src/modules/workspace/`
- 实现要点：
  - 展示工作区名称或路径摘要。
  - 展示目录树。
  - 支持当前项高亮。
  - 支持长列表滚动。
- 交付物：
  - 左侧索引区组件。
- 验收标准：
  - 工作区加载后可以浏览并看到所有索引项。
- 验证方式：
  - 手动使用多级目录样例验证。
- 风险点：
  - 大量节点一次性渲染导致侧栏卡顿。

#### T11：实现索引点击打开入口
- 目标：用户点击索引项后，能触发文档打开流程。
- 前置依赖：T10。
- 建议修改位置：
  - `src/components/sidebar/`
  - `src/modules/document/`
  - `src/stores/`
- 实现要点：
  - 文件节点可点击，目录节点仅展开或折叠。
  - 点击前先处理未保存确认逻辑占位。
- 交付物：
  - 索引点击事件处理逻辑。
- 验收标准：
  - 点击文件节点后，会尝试打开对应文档。
- 验证方式：
  - 手动点击多个索引项验证。
- 风险点：
  - 目录与文件节点行为没有区分清楚。

#### T12：实现工作区文件变化监听
- 目标：在工作区文件变动后更新索引。
- 前置依赖：T09。
- 建议修改位置：
  - `src-tauri/src/modules/workspace/watcher.rs`
  - `src-tauri/src/commands/workspace.rs`
  - 前端事件订阅层
- 实现要点：
  - 监听新增、删除、重命名和修改事件。
  - 去抖处理高频事件。
  - 优先增量更新索引，必要时全量重建。
- 交付物：
  - 文件监听服务。
  - 前端索引刷新入口。
- 验收标准：
  - 工作区文件变化后，索引能更新。
- 验证方式：
  - 手动在工作区中新增、删除、重命名 Markdown 文件。
- 风险点：
  - 不同平台 watcher 事件行为差异较大。

### 5.3 阶段三：文档读写与编辑能力

#### T13：实现文档读取命令
- 目标：Rust 侧支持按路径读取 Markdown 文件内容。
- 前置依赖：T04、T09。
- 建议修改位置：
  - `src-tauri/src/modules/document/reader.rs`
  - `src-tauri/src/commands/document.rs`
- 实现要点：
  - 校验路径有效性。
  - 读取文本内容。
  - 处理编码异常、文件不存在、权限不足等错误。
- 交付物：
  - `open_document` 命令。
- 验收标准：
  - 传入路径后能返回文档内容或结构化错误。
- 验证方式：
  - Rust 单元测试。
  - 手动打开真实 Markdown 文件。
- 风险点：
  - 编码问题导致部分文本读取异常。

#### T14：实现当前文档状态结构
- 目标：前端建立当前文档状态的统一模型。
- 前置依赖：T04、T13。
- 建议修改位置：
  - `src/types/`
  - `src/modules/document/`
  - `src/stores/`
- 实现要点：
  - 记录路径、标题、内容、最近保存内容、未保存状态、保存中状态。
  - 新建文档需要支持临时状态。
- 交付物：
  - `DocumentState`。
  - 文档状态 store。
- 验收标准：
  - 打开文档后，相关状态完整可追踪。
- 验证方式：
  - 手动调试和简单状态测试。
- 风险点：
  - 未保存状态判定逻辑不稳定。

#### T15：实现文档打开流程
- 目标：把索引点击与文档读取联通到内容展示。
- 前置依赖：T11、T13、T14。
- 建议修改位置：
  - `src/modules/document/`
  - `src/stores/`
  - `src/components/editor/`
- 实现要点：
  - 点击索引项调用打开命令。
  - 成功后写入当前文档状态。
  - 失败后展示错误反馈。
- 交付物：
  - 文档打开 action。
- 验收标准：
  - 点击索引项后能看到对应文档内容。
- 验证方式：
  - 手动切换多个文档。
- 风险点：
  - 快速连续点击导致打开结果串台。

#### T16：集成基础编辑器
- 目标：引入源码编辑区并支持基础输入。
- 前置依赖：T14、T15。
- 建议修改位置：
  - `src/components/editor/`
  - `src/modules/document/`
- 实现要点：
  - 支持文本输入、删除、复制、粘贴。
  - 支持撤销与重做。
  - 内容变化后同步更新当前文档状态。
- 交付物：
  - 编辑器组件。
- 验收标准：
  - 可以在文档中正常编辑文本。
- 验证方式：
  - 手动编辑、撤销、重做、粘贴测试。
- 风险点：
  - 编辑器选型不当导致长文本性能不足。

#### T17：实现未保存状态判定
- 目标：在编辑内容变化后正确标记文档是否未保存。
- 前置依赖：T14、T16。
- 建议修改位置：
  - `src/modules/document/`
  - `src/stores/`
  - `src/components/statusbar/`
- 实现要点：
  - 比较当前内容和最近保存内容。
  - 在标题、状态栏、索引项中至少一处展示未保存状态。
- 交付物：
  - 稳定的 `isDirty` 逻辑。
- 验收标准：
  - 修改后出现未保存标记，保存后消失。
- 验证方式：
  - 手动修改并保存当前文档。
- 风险点：
  - 自动格式化或换行差异导致误判。

#### T18：实现文档保存能力
- 目标：支持保存当前文档到原路径。
- 前置依赖：T13、T14、T17。

  - 提供保存命令。
  - 保存前设置保存中状态。
  - 保存成功后刷新最近保存内容和未保存状态。
  - 保存失败后保留当前编辑内容。
- 交付物：
  - `save_document` 命令和前端保存动作。
- 验收标准：
  - 编辑后可以保存到原文件。
- 验证方式：
  - 手动修改并保存，随后重新打开文件核对内容。
- 风险点：
  - 并发保存导致状态错乱。

#### T19：实现另存为能力

  - `save_document_as` 流程。
- 验收标准：
  - 可以把文档另存为新文件，且 UI 状态同步更新。
- 验证方式：
  - 手动另存为工作区内路径和外部路径。
- 风险点：

#### T20：实现新建文档能力
- 目标：支持在当前工作区中创建新文档。

  - `src/components/toolbar/`

## 9. Definition of Done

### 9.1 单任务完成定义
一个任务完成，至少应满足：
- 代码已实现。
- 相关状态或界面可实际触发。
- 对应异常路径至少考虑了最基本处理。
- 有可执行的验证方式。
- 与现有需求、交互、技术设计口径一致。

### 9.2 阶段完成定义
一个阶段完成，至少应满足：
- 该阶段所有 P0/P1 任务都已落地。
- 前置阶段未遗留阻断问题。
- 可以进行下一阶段联调而不依赖大量手工绕行。

### 9.3 MVP 完成定义
MVP 完成，至少应满足：
- 已达到第 3 节的 MVP 完成判定。
- 第 8 节里程碑四达成。
- 第 11 节验收清单中的核心项均有证据。

## 10. 风险清单与处理建议
- 风险 1：前端编辑器选型不适合长文档。
  - 建议：在 T16 阶段尽早用超长文档样例压测，不要等功能堆满后再发现问题。
- 风险 2：文件监听跨平台行为不一致。
  - 建议：先保证“可全量重建”的兜底方案，再做增量优化。
- 风险 3：无感渲染模式定义过宽，导致实现范围膨胀。
  - 建议：MVP 只要求“以渲染结果为主的阅读/轻编辑体验”，不要提前承诺完整富文本编辑。
- 风险 4：相对路径解析边界复杂。
  - 建议：统一基于当前文档目录和工作区根路径做解析，避免多处自行拼路径。
- 风险 5：公式与图片过多导致渲染阻塞。
  - 建议：优先做异步加载、错误占位和首屏优先策略。
- 风险 6：状态过于分散导致模式切换丢失上下文。
  - 建议：document store 作为单一事实源，视图层只消费状态，不复制状态。
