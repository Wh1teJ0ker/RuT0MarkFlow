# 技术设计文档

## 1. 文档目标
- 说明 Markdown 编辑器 MVP 的整体技术方案。
- 为 `v0.1.0` 已完成基线之后的补丁级工程优化提供延续设计依据。
- 明确 Rust、Tauri 与前端界面的职责划分。
- 明确工作区、索引、渲染、状态管理和性能优化的实现方式。
- 为后续开发任务拆解、编码实现、联调和验收提供依据。

## 2. 设计原则
- 优先完成 MVP 闭环，不为未来扩展过度设计。
- 优先复用 Tauri 提供的桌面能力，减少绕行方案。
- 文件系统访问统一收口到 Rust 层。
- 富交互与渲染展示优先放在前端层实现。
- 数据结构尽量稳定、简单、可序列化。
- 超长文档性能问题优先通过分层缓存、节流和分段渲染解决，而不是一次性引入过重架构。
- 允许在极端场景下降级显示，但不允许界面长期失去响应。

## 3. 技术选型

### 3.1 核心技术栈
- 桌面应用框架：Tauri
- 后端语言：Rust
- 前端界面：Web 技术栈
- Markdown 渲染：前端渲染引擎为主，必要时结合 Rust 侧元信息处理
- 数学公式渲染：支持 LaTeX 公式渲染能力的前端方案
- 状态持久化：本地轻量配置存储
- 前端图标：统一使用 `lucide-react` 矢量图标库，按需引入，禁止使用 emoji 表情作为功能图标

### 3.2 选型理由
- Rust 适合处理本地文件读写、索引构建、目录扫描、文件监听和性能敏感逻辑。
- Tauri 适合构建轻量桌面应用，并提供前后端桥接能力。
- 前端界面适合承载编辑器、模式切换、预览渲染、状态提示和复杂交互。
- Markdown 和公式渲染更适合在前端完成展示，以便提升交互灵活性和渲染反馈速度。
- 状态持久化只需要保存最近工作区、最近文档、主题和模式，不需要引入重型数据库。

## 4. 总体架构

### 4.1 架构分层
系统分为三层：
- 桌面壳层
- Rust 能力层
- 前端交互层

### 4.2 各层职责

#### 桌面壳层
- 提供应用窗口。
- 提供菜单、快捷键、对话框、文件夹选择等桌面能力入口。
- 承载前端资源并连接 Rust 命令能力。

#### Rust 能力层
- 处理工作区文件夹选择后的路径校验。
- 扫描工作区并构建 Markdown 索引。
- 处理文件读取、保存、另存为、新建等本地文件操作。
- 处理文件变化监听。
- 维护工作区级缓存与基础元信息。
- 统一返回结构化错误信息。

#### 前端交互层
- 渲染主界面。
- 展示工作区索引列表。
- 展示编辑区和预览区。
- 管理双模式切换。
- 管理当前文档状态、工作区状态和提示信息。
- 渲染 Markdown、图片、链接和数学公式。
- 处理快捷键和交互反馈。

## 6. 核心模块设计

### 6.1 工作区模块
职责：
- 管理当前工作区路径。
- 加载工作区基础信息。
- 管理工作区切换。
- 在应用启动时恢复最近工作区。

输入：
- 用户选择的本地文件夹路径。

输出：
- 当前工作区状态。
- 当前工作区的索引数据。
- 当前工作区可访问性结果。

关键设计：
- 工作区切换前需要检测未保存状态，由前端先做交互确认。
- Rust 层只负责执行新的工作区加载，不负责业务确认弹窗。
- 工作区路径需要规范化，避免相对路径或大小写差异导致状态错乱。

### 6.2 索引模块
职责：
- 扫描工作区内 Markdown 文件。
- 识别子目录结构。
- 建立索引数据。
- 响应文件新增、删除、重命名和修改事件。
- 向前端提供可展示的索引列表或树结构。

索引建议包含的信息：
- 文件绝对路径
- 工作区相对路径
- 文件名
- 所属目录
- 文件扩展名
- 修改时间
- 文件大小
- 是否当前存在

设计要求：
- 索引建立过程应支持后台执行。
- 索引更新优先增量化。
- 索引数据应适合前端直接消费。
- 工作区文件数量较多时，索引更新不应阻塞界面。
- 只收录 `.md` 和 `.markdown` 文件。

### 6.3 文档读写模块
职责：
- 读取 Markdown 文件内容。
- 保存当前文件。
- 执行另存为。
- 创建新文档。
- 处理读写失败和路径异常情况。

设计要求：
- 读写操作通过 Rust 层统一处理。
- 前端不直接进行任意本地文件系统访问。
- 保存结果应明确返回成功或失败状态。
- 新建文档流程需要区分“内存中新建未落盘”和“实际创建文件”两个阶段。
- 写入前需要校验目标路径是否仍位于当前工作区内，除非明确执行另存为到外部位置。

### 6.4 编辑器模块
职责：
- 承载源码输入。
- 管理撤销、重做、复制、粘贴等基本编辑能力。
- 维护当前文档未保存状态。
- 在双栏模式下与预览区联动。
- 在无感编辑模式（immersive-preview）下承载 `contentEditable` 编辑态（`ImmersiveEditor`）。

设计要求：
- 编辑器组件应支持长文本编辑。
- 编辑器组件应支持较好的滚动性能。
- 编辑器与渲染结果之间需要低延迟同步。
- 编辑器内容更新与磁盘保存状态应区分管理，不应用“文本变化”和“保存成功”混为同一个状态源。

### 6.5 渲染模块
职责：
- 将 Markdown 内容渲染为可读视图。
- 支持无感 Markdown 渲染模式。
- 支持双栏模式下的预览区域渲染。
- 解析并展示相对路径图片、相对链接和公式内容。

渲染支持范围：
- 标题、段落、列表、引用、代码块、表格、链接、图片
- 行内公式
- 块级公式
- 常见 LaTeX 数学表达式

设计要求：
- 两种模式下的渲染结果尽量一致。
- 渲染失败时应允许降级展示。
- 图片或公式渲染异常时应返回前端可识别状态。
- Markdown 解析和公式渲染应拆分阶段，避免单个公式失败影响整篇文档渲染。

### 6.6 状态管理模块
职责：
- 维护当前工作区状态。
- 维护当前文档路径、内容和保存状态。
- 维护当前模式状态。
- 维护索引更新状态、错误提示状态和加载状态。
- 维护最近使用设置的恢复状态。

设计要求：
- 状态结构需要清晰分层。
- 文档状态与工作区状态应解耦。
- 模式切换不应破坏当前文档状态。
- 临时 UI 状态与持久化设置状态应分开管理。

## 8. Tauri 命令接口设计

> 以下命令对应 `src-tauri/src/lib.rs` 中 `generate_handler!` 注册的实际命令，签名与 `commands/` 下各模块一致。

### 8.1 工作区相关命令
- `select_workspace(app: AppHandle) -> CommandResponse<WorkspaceLoadResult>` — 打开系统文件夹选择器，扫描并加载工作区，成功后写入 AppState.authorized_workspace
- `load_workspace(app: AppHandle, root_path: String) -> CommandResponse<WorkspaceLoadResult>` — 按指定路径加载工作区（用于恢复最近工作区），成功后写入 AppState.authorized_workspace
- `refresh_workspace_index(app: AppHandle) -> CommandResponse<WorkspaceLoadResult>` — 重新扫描并刷新当前工作区索引，root_path 从 AppState 读取
- `start_workspace_watcher(app: AppHandle) -> CommandResponse<String>` — 启动文件监听，root_path 从 AppState 读取
- `stop_workspace_watcher(app: AppHandle) -> CommandResponse<String>` — 停止文件监听

### 8.2 文档相关命令
- `open_document(app: AppHandle, relative_path: String) -> CommandResponse<DocumentOpenResult>` — 读取工作区内 Markdown 文件，root_path 从 AppState 读取
- `save_document(app: AppHandle, relative_path: String, content: String) -> CommandResponse<DocumentSaveResult>` — 覆写保存（路径须在工作区内），root_path 从 AppState 读取
- `pick_save_path(app: AppHandle, default_name: String) -> CommandResponse<PickSavePathResult>` — 打开系统保存对话框，返回用户选择的目标路径信息与一次性 `saveToken`，root_path 从 AppState 读取
- `save_document_as(app: AppHandle, save_token: String, content: String) -> CommandResponse<DocumentSaveResult>` — 保存到 picker 选定路径（另存为）。前端不再传入 `target_path`，而是传入 `pick_save_path` 颁发的一次性 `save_token`；Rust 侧消费该 token 并从 AppState 取回绑定路径后写入。token 单次有效，使用后失效；root_path 从 AppState 读取

### 8.3 设置与状态相关命令
- `load_app_settings(app: AppHandle) -> CommandResponse<AppSettings>` — 读取持久化设置
- `save_app_settings(app: AppHandle, settings: AppSettings) -> CommandResponse<()>` — 写入持久化设置
- `set_document_dirty(state: State<DocumentDirtyState>, dirty: bool) -> ()` — 同步前端 dirty 状态到 Rust，用于条件 prevent_close

### 8.4 健康检查与更新相关命令
- `health_check() -> CommandResponse<String>` — 返回健康检查摘要字符串
- `version_info() -> CommandResponse<VersionInfo>` — 返回版本清单信息
- `check_for_updates(app: AppHandle) -> CommandResponse<UpdateCheckResult>` — 检查 GitHub Release 更新
- `install_update(app: AppHandle) -> CommandResponse<UpdateInstallResult>` — 下载并安装更新

### 8.5 结果结构
`WorkspaceLoadResult` 包含：
- `workspace`
- `indexTree`
- `flatEntries`
- `restoredDocumentPath`

`DocumentOpenResult` 包含：
- `path`
- `relativePath`
- `content`
- `updatedAt`

`DocumentSaveResult` 包含：
- `path`
- `updatedAt`
- `contentHash`

`PickSavePathResult` 包含：
- `saveToken`（一次性令牌，绑定 picker 选定路径，供 `save_document_as` 消费，单次有效）
- `absolutePath`
- `relativePath`（可选，目标在工作区内时才有值）
- `isWithinWorkspace`

## 9. 前端状态结构设计

### 9.1 顶层状态切分
建议至少拆分为以下状态域：
- `workspaceStore`
- `documentStore`
- `renderStore`
- `settingsStore`
- `uiStore`

### 9.2 workspaceStore
负责：
- 当前工作区信息
- 索引树数据
- 扁平索引数据
- 索引加载状态
- 索引更新状态
- 最近一次索引错误

### 9.3 documentStore
负责：
- 当前文档路径
- 当前文档内容
- 最近成功保存内容
- 未保存状态
- 新建文档临时状态
- 保存中状态
- 打开失败或保存失败错误

### 9.4 renderStore
负责：
- 当前渲染 HTML
- 渲染中状态
- 渲染降级标记
- 图片失败列表
- 公式失败列表
- 渲染缓存键

### 9.5 settingsStore
负责：
- 当前主题
- 当前模式
- 最近工作区路径
- 最近文档路径

### 9.6 uiStore
负责：
- 当前弹窗状态
- Toast 或消息提示
- 当前是否展示未保存确认框
- 当前是否展示错误对话框

## 10. 关键数据流设计

### 10.1 打开工作区数据流
1. 前端触发工作区选择。
2. Tauri 打开系统文件夹选择器。
3. Rust 获取路径并进行可访问性校验。
4. Rust 启动工作区扫描并构建索引。
5. Rust 返回工作区信息和索引数据。
6. 前端写入 `workspaceStore`。
7. 前端根据设置尝试恢复最近文档。

### 10.2 打开文档数据流
1. 用户点击索引项。
2. 前端检查当前文档是否存在未保存修改。
3. 若需要确认，先完成确认弹窗流程。
4. 前端调用 `open_document`。
5. Rust 读取文件内容并返回。
6. 前端写入 `documentStore`。
7. 前端触发渲染流程，更新 `renderStore`。

### 10.3 编辑与预览联动数据流
1. 用户修改编辑器内容。
2. 前端立即更新 `documentStore.content`。
3. 前端基于 `content !== lastSavedContent` 更新 `isDirty`。
4. 前端对渲染任务做节流调度。
5. 渲染模块解析 Markdown、处理链接和图片、执行公式渲染。
6. 渲染结果写入 `renderStore`。
7. 页面刷新预览区域或无感渲染区域。

### 10.4 保存数据流
1. 用户发起保存。
2. 前端校验当前文档是否具备保存路径。
3. 若为新建未落盘文档，则先触发另存为或创建路径流程：前端调用 `pick_save_path`，Rust 打开系统保存对话框并颁发一次性 `saveToken`（绑定到 picker 选定路径）。
4. 前端调用 `save_document` 或 `save_document_as`（对另存为传入 `saveToken`，不再传入裸 `target_path`）。
5. Rust 消费 `saveToken` 并从 AppState 取回绑定路径后执行文件写入；token 使用后失效。
6. Rust 返回保存结果。
7. 前端更新 `lastSavedContent`、`isDirty`、`isSaving`。
8. 状态栏或消息提示更新为保存成功或保存失败。

### 10.5 文件变化同步数据流
1. Rust 监听工作区文件变化。
2. 检测到新增、删除、重命名或修改事件。
3. Rust 更新索引数据。
4. 通过事件或命令轮询通知前端。
5. 前端更新索引区状态。
6. 若当前文档失效，则触发前端提示和状态修正。

## 11. 模式设计

### 11.1 无感编辑模式
- 前端以 `contentEditable` 容器替代 `dangerouslySetInnerHTML` 只读渲染，用户可直接在渲染态编辑文本（类 Typora 无感编辑）。
- 编辑时以 Markdown 源码为唯一事实源：用户在渲染态编辑 → 反序列化 DOM 为 Markdown 源码 → 更新 documentStore.content → dirty 传播。
- 光标保持：编辑后重新渲染时尽量保持光标位置（DOM 节点标记 + Selection API）。
- 语法块（代码块、公式块）在无感编辑模式下以可编辑源码块呈现：代码块本身即为渲染态源码，可直接编辑；公式块在 `postprocessMath` 中以 `.math-source-block` 容器包裹 KaTeX 输出并携带 `data-formula-source`/`data-formula-display`，反序列化时以源码回写（不实现公式本身的 WYSIWYG 编辑，符合 out_of_scope）。
- 保留 DOMPurify 净化管线，渲染输出仍经过 sanitize。
- 不引入 ProseMirror、Lexical、TipTap 等富文本编辑框架，保持轻量。

#### 11.1.1 ImmersiveEditor 组件
- 新增 `src/components/content/ImmersiveEditor.tsx`：承载沉浸式预览区的 `contentEditable` 编辑容器。
- `ContentArea` 的 `immersive-preview` 分支由原 `dangerouslySetInnerHTML` 的 `<div>` 切换为 `<ImmersiveEditor>`；split-editor 分支保持不变（仍在 `out_of_scope` 内）。
- 容器 `innerHTML` 完全由组件 effect 管理（不通过 React children 渲染），避免 React 调和与 contentEditable 冲突；`renderedHtml`（已由渲染管线 marked → DOMPurify → KaTeX 净化）经 ref 写入容器。
- 编辑提取：`onInput` 时调用 `deserializeHtmlToMarkdown`（`src/services/render/serialize.ts`）遍历 contentEditable DOM 重建 Markdown 源码，调用 `onContentChange` 同步到 `documentStore.content`。支持标题 H1–H6、段落、列表（含嵌套）、引用、代码块、表格、粗体/斜体/删除线、行内代码、链接/图片；公式块以携带源码的 `.math-source-block` 容器参与回写。
- 自身编辑回流不重置 DOM：组件以 `lastEmittedContentRef` + `hasEmittedRef` 判定 `content` 是否为自身刚发出的内容（尾随空白容差），若是则跳过 `innerHTML` 重置（保持光标稳定）；仅文档切换 / 撤销重做 / 外部重置等非自身编辑触发时才以新 `renderedHtml` 重置。
- 渲染中不替换 DOM：`isRenderPending` 期间 effect 不清空、不替换 contentEditable 容器内容，仅在容器上设置 `data-render-pending="true"` 驱动 CSS overlay 显示加载提示，避免长文档异步分批渲染时丢失用户正在编辑的输入与光标。
- 光标位置在重置前后尽量保持：重置前用 `Range.selectNodeContents + setEnd` 测量选区在容器内的字符偏移 `{start,end}`，重置后下一帧用 `TreeWalker` 遍历文本节点重新定位并恢复 Selection。
- 链接委托与 `ContentArea` 一致：内部 `.md` 链接（`data-internal-md`）走 `onOpenDocument`，外链（`data-external-link`）走 Tauri `openUrl`；上层 `ContentArea` 的 `handlePreviewClick`（图片错误重试等）以 `onClick` prop 注入。
- 粘贴仅保留纯文本：`onPaste` 拦截富文本粘贴，通过 Selection API（`Range.insertNode` 插入文本节点）插入纯文本，`execCommand("insertText")` 作为回退，避免未净化 HTML 进入容器。
- 已知限制：公式块以渲染态呈现，用户直接编辑 KaTeX DOM 不会被反序列化捕获；公式的可编辑事实源是容器上的 `data-formula-source`。这是“不实现公式 WYSIWYG”约束的直接结果，符合 acceptance_criteria。

#### 11.1.2 DOM → Markdown 反序列化
- 新增 `src/services/render/serialize.ts`：`deserializeHtmlToMarkdown(root)` 遍历渲染态 DOM 重建 Markdown 源码，纯 DOM walk，不依赖 marked、不修改 marked 配置。
- 节点映射：`h1–h6 → # 语法`、`p → 段落`、`ul/ol → -/1.`（含 `  ` 缩进的嵌套）、`blockquote → > `、`pre>code → ```fenced```（从 `language-*` class 提取语言）`、`table → GFM 管道表（表头 + `---` 分隔 + 行）`、`hr → ---`、`strong/b → **`、`em/i → *`、`del/s/strike → ~~`、`code → ` 反引号（内容含反引号时加空格并选更长 fence）`、`a → [text](href)`、`img → ![alt](src)`（优先 `data-original-src`）、`br → 硬换行`。
- 公式反序列化：`.math-source-block` 容器读取 `data-formula-source` 与 `data-formula-display`，重建 `$$...$$`（块级）/ `$...$`（行内）。
- 输出为逻辑块序列，以单空行分隔，多行块（列表/表格/引用/代码围栏）内部 `\n` 保留；末尾以换行终止。
- `postprocessMath`（`src/services/render/math.ts`）同步调整：成功渲染的 KaTeX 输出以 `div`（块级）/`span`（行内）`.math-source-block` 包裹，写入 `data-formula-source`（属性值经转义）与 `data-formula-display`；失败公式仍以 `.math-error` 回退，行为不变。
- 单测：`src/services/render/__tests__/serialize.test.ts` 覆盖各节点类型的往返；`math.test.ts` 增补源码包裹属性验证；`immersive-editor.test.tsx` 增补 onInput 反序列化保留语法标记的验证。

### 11.2 双栏编辑模式
- 左侧展示源码编辑器。
- 右侧展示 Markdown 渲染结果。
- 需要保证源码输入与预览结果同步。
- 两栏宽度可通过 ResizeHandle 组件拖拽调整，比例持久化到 AppSettings。
- 该模式作为 MVP 的主要稳定编辑模式。

### 11.3 模式切换策略
- 模式状态由前端统一管理。
- 切换模式时复用当前文档状态。
- 不重复触发不必要的文件读取。
- 切换时优先复用已有渲染结果或缓存结果。
- 模式切换只影响视图层，不重置编辑器内容和文档状态。

## 12. Markdown 与公式渲染设计

### 12.1 Markdown 渲染原则
- 优先支持常用 Markdown 语法。
- 对扩展语法保持可降级展示。
- 保持渲染结果稳定可预测。
- 在工作区语境下处理相对路径资源。
- 渲染结构尽量语义化，便于后续优化样式和交互。

### 12.2 渲染流水线
1. 获取原始 Markdown 字符串。
2. 进行内容哈希计算，判断是否命中渲染缓存。
3. 执行 Markdown 解析。
4. 处理代码块、表格、链接、图片节点。
5. 识别并渲染公式节点。
6. 生成 HTML 结果。
7. 记录图片错误、公式错误、降级块信息。
8. 推送结果到前端视图。

### 12.3 图片与链接处理
- 相对路径图片基于当前文档所在目录解析。
- `../` 段被词法折叠，但最终解析路径不得逃逸出工作区根边界；逃逸尝试与绝对文件路径（`/foo`、`C:\foo`）不加载，输出错误占位（防御纵深：Rust 端 asset scope 与 `read_markdown_file` 同样强制边界）。
- 相对路径链接优先解析为工作区内资源。
- 指向工作区内 Markdown 文件的相对链接基于当前文档目录解析为工作区根相对路径（`data-internal-md`），交由应用内打开动作调用；`..` 逃逸被钳制到根，Rust 端再做一次 `canonicalize` + `starts_with` 边界校验。
- 链接的 `#anchor` 片段保留在 `data-internal-md-anchor` 属性中，不丢失；纯 `#anchor` 锚点链接保持不变，交给 webview 默认行为。
- 指向外部 URL 的链接由用户确认后使用系统默认方式打开。
- 图片加载失败时返回错误状态给前端展示。

### 12.4 公式渲染设计
- 支持行内公式。
- 支持块级公式。
- 使用可渲染 LaTeX 数学表达式的前端渲染方案。
- 公式渲染失败时保留原始文本或展示明确反馈。
- 公式渲染需要兼顾浅色与深色主题可读性。
- 公式渲染应尽量按节点执行，避免整篇内容因单个公式失败而全部回退。

## 13. 性能设计

### 13.1 性能关注点
- 工作区扫描性能
- 索引建立与更新性能
- 文档打开性能
- Markdown 渲染性能
- 超长 Markdown 文档显示性能
- 滚动性能
- 模式切换性能

### 13.2 超长 Markdown 文档优化策略
- 避免每次输入都进行全量高成本渲染。
- 对渲染更新进行节流或批量刷新控制。
- 对超长内容的显示区域采用分段渲染或虚拟化策略。
- 优先保证可视区域渲染速度。
- 模式切换时优先复用已有解析结果。
- 长文档滚动时避免重复计算整个文档布局。
- 对图片和公式较多的文档采用异步补全策略。

### 13.3 索引性能优化策略
- 初次扫描采用后台任务执行。
- 文件变化后优先进行增量更新。
- 索引结构同时保留扁平结构和树结构，减少前端重复计算。
- 索引数据可结合时间戳或摘要做轻量缓存。

### 13.4 渲染性能优化策略
- 对相同内容的渲染结果可进行缓存。
- 公式渲染应避免在无变化时重复执行。
- 图片资源加载异步处理。
- 渲染异常不应阻塞主交互流程。
- 在超长文档场景中，允许先渲染首屏和基础结构，再补全其他块。

### 13.5 MVP 性能分阶段落地建议
- 第一阶段：实现基础节流和渲染缓存。
- 第二阶段：补充超长文档分段渲染。
- 第三阶段：在确有必要时引入更细粒度虚拟化。

## 14. 状态持久化设计
- 持久化最近工作区路径。
- 持久化最近打开文档路径。
- 持久化当前使用模式。
- 持久化主题设置。

设计要求：
- 状态持久化数据应尽量简单。
- 持久化失败不应影响主流程。
- 启动恢复逻辑应允许用户重新选择工作区。
- 若最近工作区不存在或不可访问，应安全退回空工作区状态。

## 15. 错误处理设计

### 15.1 主要错误类型
- 工作区不可访问
- 文件不存在
- 文件读取失败
- 文件保存失败
- 索引建立失败
- 图片资源不可读取
- 公式渲染失败
- 渲染超时或性能退化
- 文档链接目标不存在

### 15.2 错误处理要求
- Rust 层返回结构化错误信息。
- 前端将错误转换为可理解提示。
- 可恢复错误不应导致当前应用退出。
- 对超长文档性能退化场景，应允许降级处理而非卡死界面。
- 保存失败时必须保留内存中的当前编辑内容。
- 工作区切换失败时应保留原工作区状态，避免界面进入不一致状态。

## 16. 安全与边界设计
- 应用默认只处理用户主动选择的本地工作区。
- 文件访问范围以当前工作区和用户主动打开的文件为主。
- 不主动上传用户文档内容。
- 外部链接打开行为应明确区分应用内跳转和系统外部跳转。
- 默认不执行 Markdown 中的脚本内容。
- 对原始 HTML 渲染需采用保守策略，避免引入不必要的执行面。

### 16.1 工作区授权模型
- Rust `AppState` 持有 `authorized_workspace: Mutex<Option<PathBuf>>`，保存用户通过 picker 授权的工作区根路径。
- `select_workspace` 和 `load_workspace` 成功后写入 AppState。
- `open_document`、`save_document`、`save_document_as`、`refresh_workspace_index` 等命令从 AppState 读取 root_path，忽略前端传入值。
- 所有文件操作经 `canonicalize` + 边界检查确保在授权 workspace 内。
- 前端命令签名移除 root_path 参数（或保留但后端忽略）。

### 16.2 CSP 与资源访问
- 启用严格 CSP，禁止 `unsafe-inline` 和 `unsafe-eval`。当前策略：
  - `default-src 'self'`
  - `img-src 'self' asset: http://asset.localhost https://asset.localhost data:`
  - `media-src` 同 `img-src`
  - `style-src 'self' 'unsafe-inline'`（KaTeX 内联样式所需，且不影响脚本执行面）
  - `script-src 'self'`、`font-src 'self' data:`、`connect-src 'self' ipc: http://ipc.localhost`
  - `frame-src 'none'`、`object-src 'none'`、`base-uri 'self'`
- 移除 `$HOME/**` 静态 asset scope，`assetProtocol.scope` 收窄到 `["$APPDATA/**", "$RESOURCE/**"]`，并显式 `enable: true`。`$HOME/**` 不再全量放行。
- **运行时动态 scope 注册（替代 `$HOME/**`）**：`select_workspace` / `load_workspace` 在 `AppState::authorize` 成功后，调用 `app.asset_protocol_scope().allow_directory(canonical_root, true)` 将规范化的工作区根路径递归注册到 Tauri asset 协议作用域。授权工作区内的图片资源由前端 `convertFileSrc` 转换为 `asset://` URL，再由 Tauri 资源协议受控读取。该机制保证：静态 scope 不含 `$HOME` 通配，只有用户主动授权过的工作区目录可通过 `asset://` 加载。
  - 注意：`allow_directory` 的运行时拦截依赖 Tauri 运行时与 webview，无法用纯单元测试覆盖；`AppState::authorize` 返回的规范化路径即注册到 scope 的路径（契约由 `app_state.rs` 单元测试固定），实际 `asset://` 加载验证需手动 / 集成测试覆盖。
- 图片资源由后端受控读取，通过 `convertFileSrc` 转换为受控 asset URL。
- 图片错误占位使用 DOM API + `textContent`，不拼接 `outerHTML`。`resource.ts` 不再输出内联 `onerror` 属性（避免 `unsafe-inline` 与二次 HTML 注入）；运行时 `<img>` 加载失败由预览区委托 `error` 监听器（`ContentArea`）用 DOM API 替换为占位 span。解析期不可解析的图片输出转义后的占位 span。

### 16.3 原子写入与 symlink 防护
- `write_markdown_file` 和 `write_to_any_path` 改为临时文件 + `fsync` + 原子 `rename`：在目标同目录写入 `.{filename}.tmp.{random}`，`sync_all`（fsync / FlushFileBuffers）后再 `rename` 覆盖目标；写入或 rename 失败时清理临时文件，原文件不被截断。
- symlink 不跟随：在 `canonicalize` 之前用 `symlink_metadata` 检查目标路径，遇到 symlink 直接返回错误（"目标路径是符号链接，拒绝写入"），避免 `canonicalize` 跟随链接后绕过检查。`validate_and_write` 内再做一次深度防御检查。
- Windows 上 `std::fs::rename` 覆盖已存在文件会失败，采用 try-rename → 失败回退 `remove_file` + `rename` 的策略。
- 保存/另存为使用内容快照（`contentToSave`），避免竞态将未落盘内容标记为已保存。
