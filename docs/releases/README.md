# 后续版本文档索引

## 1. 目录定位
- `docs/releases/` 用于承接 `v0.1.0` 之后的版本规划和变更记录。
- 根目录 `docs/01` 到 `docs/05` 保持为 MVP 基线文档，不继续混入后续版本细节。
- 每个版本固定维护两份文档：
  - `规划需求.md`：定义该版本的目标、范围、验收标准和不做项。
  - `更新日志.md`：记录该版本的实际实施状态、关键变更和验证结果。
- 跨版本的演进主线与节奏由 `docs/releases/路线规划.md` 统一约束。

## 2. 当前规划版本
- 当前有一个激活中的后续版本：`v0.1.2`。
- `v0.1.1`：稳定化 patch，已完成验收。
- `v0.1.2`：Windows 跨平台修复（活跃实施中）。
- `v0.2.0`：写作体验增强（minor），已完成规划，尚未进入实施或发布。
- 后续 `v0.3.0` / `v0.4.0` / `v0.5.0` 的主线与范围概要见 `路线规划.md`，待各自启动时再建立对应 `规划需求.md`。

## 3. 当前目录结构
```text
docs/releases/
  README.md
  路线规划.md
  v0.1.1/
    规划需求.md
    更新日志.md
  v0.1.2/
    规划需求.md
    更新日志.md
  v0.2.0/
    规划需求.md
    更新日志.md
```

说明：
- 当前索引列出已有实施或规划入口的版本。
- 若未来需要新增 patch 版本或启动下一个 minor 版本，应先新增对应 `规划需求.md`，再纳入这里的目录结构。
- `路线规划.md` 是所有后续版本的上游依据，单版本范围不得与之冲突。

## 4. 维护规则
- 每个版本实施阶段只更新该版本 `规划需求.md` 与 `更新日志.md`。
- 若新增版本，先补齐对应 `规划需求.md`，再开始代码和发布工作。
- 任何当前实施版本发生实际变更后，同步更新对应 `更新日志.md`。
- 若版本完成验收，再将结论回写 `docs/05-验收记录.md`。
- 任何主线方向调整，先更新 `路线规划.md`，再调整对应版本规划。
