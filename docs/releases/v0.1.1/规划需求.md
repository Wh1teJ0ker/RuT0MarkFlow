# v0.1.1 规划需求

## 1. 版本定位
- 版本号：`v0.1.1`
- 版本类型：patch
- 目标：作为 `v0.1.0` 之后唯一激活的稳定化版本，一次性收口发布、可诊断性、恢复链路和回归门禁问题。

## 2. 本版本要解决的问题
- 发布前版本号、Tag、构建产物、图标和远程 Release 资产之间仍可能出现不一致。
- README 过于简略，缺少技术栈、编译方式、测试入口和发布产物说明。
- GitHub Release 需要在常规安装包之外新增便携版产物，并保证命名与版本一致。
- 异常提示来源分散，前端依赖字符串拼接和前缀判断，用户看到的恢复路径不统一。
- 健康检查、恢复链路、压测和仓库规范检查还需要进一步门禁化，避免只靠人工确认。

## 3. 范围
- 版本同步校验、版本展示和构建元数据收口。
- 应用图标资源对齐与发布资产命名规范化。
- `README.md` 的产品说明、技术栈、开发/编译/测试/发布说明补齐。
- GitHub Actions / Release 流水线稳定化，并新增便携版发布资产。
- 工作区、文档、保存、恢复相关异常提示与恢复动作统一。
- 健康检查、恢复链路回归、压测入口和仓库规范扫描的门禁化。
- 文档同步：`docs/releases/`、`docs/04-开发任务清单.md`、`docs/05-验收记录.md`、必要的技术设计与版本规则文档。

## 4. 明确不做
- 全文搜索
- 导出能力
- 插件系统
- AI 能力
- 新的渲染模式
- Linux 全量兼容
- 与本轮稳定化无关的大型重构

## 5. 验收标准
- 版本信息在 `package.json`、`src-tauri/Cargo.toml`、`src-tauri/tauri.conf.json` 与 `version-manifest.json` 中保持同步，并能通过 `pnpm version:check -- --tag v0.1.1`。
- `README.md` 至少包含产品摘要、技术栈、本地开发方式、构建/编译方式、测试命令和 Release 产物说明。
- GitHub Release 除常规构建产物外，还能产出命名一致的便携版资产。
- 关键异常提示保留结构化错误来源，不再依赖 UI 侧的字符串前缀判断来决定恢复动作。
- 工作区恢复、文档打开失败、保存失败等场景对用户提供统一的恢复入口与文案口径。
- 健康检查、恢复链路回归、压测和仓库规范检查具备可重复执行的验证命令或工作流门禁。
- `pnpm test`、`pnpm build`、`cargo test --manifest-path src-tauri/Cargo.toml`、`cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings` 全部通过。
