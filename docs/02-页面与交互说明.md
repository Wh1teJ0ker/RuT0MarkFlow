# 页面与交互说明

## 1. 文档目标
- 说明 Markdown 编辑器 MVP 的页面结构、页面状态、操作入口、核心交互流程与异常处理方式。
- 为技术设计、组件拆分和开发任务分解提供明确的交互依据。
- 本文档只描述用户能看到和能操作的内容，不展开底层实现细节。

## 2. 产品形态
- 产品为桌面应用。
- 产品以工作区方式运行。
- 用户进入应用后，首先选择一个本地文件夹作为工作区。
- 工作区加载后，应用围绕该文件夹中的 Markdown 文档进行浏览、打开、编辑、预览、跳转和切换。
- 当前版本以 MVP 为目标，优先保证高频主流程清晰、稳定、少打断。

## 3. 页面总体结构

### 3.1 页面组成
主界面由以下区域组成：
- 顶部工具栏
- 左侧工作区索引区
- 中央内容区
- 底部状态栏

### 3.2 默认布局原则
- 页面以编辑和阅读为主，不做营销式首屏。
- 主操作入口应常驻可见，避免深层隐藏。
- 左侧索引区固定承担工作区导航职责。
- 中央内容区固定承担阅读、编辑和预览职责。
- 底部状态栏固定承担状态反馈职责。
- 在双栏模式下，左右两栏宽度可通过拖拽分隔条自由调整，但不应因内容变化频繁抖动。
- 侧边栏与内容区之间的分隔条可拖拽缩放，宽度持久化。

## 4. 顶部工具栏

### 4.1 工具栏功能项
顶部工具栏用于承载全局操作入口，建议按从左到右的顺序包含：
- 选择工作区
- 切换工作区
- 新建文档
- 保存文档
- 另存为
- 模式切换
- 主题切换

### 4.2 工具栏行为说明
- “选择工作区”在空工作区状态和正常工作状态下都应可见。
- “切换工作区”可与“选择工作区”复用同一入口，也可以在已有工作区时显示为不同文案。
- “新建文档”在已有工作区时可用；没有工作区时默认禁用或点击后提示先选择工作区。
- “保存文档”在当前文档存在且已修改时可用；没有变更时可保持禁用或显示普通状态。
- “另存为”在当前文档已打开或已有编辑内容时可用。
- “模式切换”应明确显示当前模式，不应让用户误判当前视图。
- “主题切换”至少支持浅色和深色两种状态。

### 4.3 工具栏视觉与交互要求
- 常用高频操作应在首层可见。
- 当前模式应在工具栏中可以直接识别。
- 当前工作区相关入口应始终可访问。
- 图标和文案应尽量短，避免工具栏过度拥挤。
- 按钮禁用状态应清晰可辨。
- 图标必须使用统一矢量图标库，禁止使用 emoji 表情作为功能图标。

## 5. 左侧工作区索引区

### 5.1 区域职责
左侧区域用于展示当前工作区的 Markdown 索引内容，并承担主要文档切换入口。

### 5.2 展示信息
应包含以下内容：
- 当前工作区名称或路径摘要
- Markdown 文件列表
- 子目录层级关系
- 当前选中文档状态
- 索引加载状态或更新状态

### 5.3 索引项行为
- 单击索引项直接打开文档。
- 当前打开文档在索引中应有高亮状态。
- 未保存文档在索引中可追加轻量标记，例如圆点或星号。
- 多层目录应通过缩进、折叠或树状结构展示清楚层级。
- 文件名过长时应截断显示，但应支持悬停查看完整路径或名称。

### 5.4 索引区状态要求
- 文档很多时，索引区滚动应保持流畅。
- 超长工作区列表场景下不应出现明显卡顿。
- 索引正在建立或更新时，区域中应有轻量状态提示。
- 索引失败时，区域中应保留可恢复入口，例如重试或重新选择工作区。

## 6. 中央内容区

### 6.1 区域职责
中央内容区是阅读、编辑和预览的核心区域，需要支持两种模式，并承担文档打开后的主工作面。

### 6.2 模式一：无感编辑模式
- 以类 Typora 的无感编辑方式呈现 Markdown 内容，用户可直接在渲染态点击编辑文本。
- 用户看到的主要是格式化后的内容结构，而不是纯源码。
- 内容区域需要保持良好的阅读节奏、层级和排版。
- 标题、段落、列表、引用、代码块、表格、图片、公式等内容应清晰可辨。
- 无感编辑模式下，编辑内容实时同步到文档源码，dirty 状态正确传播。
- 代码块和公式块在无感编辑模式下以可编辑的源码块呈现，不尝试 WYSIWYG 编辑公式本身。
- 模式切换后编辑内容不丢失。

### 6.3 模式二：双栏编辑模式
- 左侧为源码编辑区。
- 右侧为 Markdown 渲染预览区。
- 编辑区和预览区并排展示。
- 编辑区负责源码输入，预览区负责结构化结果展示。
- 两栏宽度可通过拖拽分隔条自由调整，比例持久化。
- 双栏模式下应保证源码输入和预览查看可以同时进行。

### 6.4 中央内容区通用要求
- 两种模式之间可以直接切换。
- 模式切换后当前文档内容、保存状态、滚动上下文和基础编辑上下文不应丢失。
- 文档切换后内容区应快速更新。
- 文档打开失败时，应在内容区显示明确错误信息，而不是空白无反馈。
- 空文档状态应可编辑，不应出现错误渲染结果。
- 超长 Markdown 文档显示、滚动和模式切换时不应明显卡顿。
- 超长文档场景下需要优先保证可读性与交互流畅性。

## 7. 底部状态栏

### 7.1 状态栏展示内容
状态栏用于展示当前页面的核心状态信息，建议包含：
- 当前工作区路径或名称
- 当前文档名称
- 当前保存状态
- 当前模式
- 索引状态
- 渲染状态

### 7.2 状态栏要求
- 状态信息应简洁、稳定。
- 保存中、未保存、保存失败、索引更新中、渲染中等状态应可识别。
- 状态栏文本应避免过长，超长路径可做折叠展示。
- 状态变化应及时，不应出现明显滞后。

## 8. 主要页面状态

### 8.1 初始空工作区状态
- 应用首次启动时，页面进入空工作区状态。
- 中央内容区不展示具体文档内容。
- 页面应提示用户先选择一个工作区文件夹。
- 工具栏中保留选择工作区入口。
- 与当前文档相关的按钮应处于禁用或不可触发状态。

### 8.2 工作区加载状态
- 用户选择工作区后，页面进入加载状态。
- 系统开始扫描 Markdown 文件并建立索引。
- 左侧索引区可逐步展示结果，也可以在加载完成后统一展示。
- 加载过程不应阻塞整个界面。
- 中央内容区可以展示“正在加载工作区”之类的轻提示。

### 8.3 工作区加载完成但未打开文档状态
- 工作区已加载，索引已可用。
- 若尚未恢复或打开文档，中央内容区显示引导信息，例如提示从左侧选择文档或新建文档。
- 工具栏中的新建文档入口应可用。

### 8.4 文档打开状态
- 用户点击索引项后，页面进入文档打开状态。
- 当前文档应在索引区高亮。
- 内容区展示对应文档内容。
- 状态栏同步更新当前文档信息。

### 8.5 空文档状态
- 新建文档后，如果内容为空，内容区应保持可编辑。
- 空文档也应保持可切换模式。
- 空文档状态下不应出现错误渲染表现。
- 若新建文档尚未落盘，应在界面中体现其临时状态。

### 8.6 未保存状态
- 当文档被修改但尚未保存时，页面需要展示未保存状态。
- 未保存状态应能在标题、索引标记、状态栏或工具栏中至少一处明确体现。
- 用户关闭文档、切换文档、切换工作区或关闭窗口时，需要触发确认提示。

### 8.7 保存中状态
- 用户触发保存后，界面可短暂展示保存中状态。
- 保存成功后，应及时取消未保存标记。
- 保存失败后，应保留当前编辑内容并给出错误提示。

### 8.8 索引更新状态
- 当工作区文件发生变化时，系统进入索引更新状态。
- 索引区内容应同步更新。
- 若当前文档仍然有效，应尽量保持当前阅读和编辑上下文。
- 若当前文档被删除、重命名或失效，应提示用户当前文档状态变化。

### 8.9 异常状态
页面需要处理以下异常状态：
- 工作区无法访问
- 文件读取失败
- 文件保存失败
- 图片资源加载失败
- 公式渲染失败
- 索引建立失败
- 文档链接目标不存在

异常状态要求：
- 给出明确提示。
- 不应导致应用整体不可用。
- 尽量保留当前可恢复内容。
- 对可重试场景应提供明确的恢复入口。

## 9. 核心交互流程

### 9.1 选择工作区流程
1. 用户打开应用。
2. 用户点击“选择工作区”。
3. 系统弹出本地文件夹选择器。
4. 用户选择本地文件夹。
5. 系统开始扫描文件夹并建立索引。
6. 页面展示工作区索引列表。
7. 若存在最近文档且仍有效，系统可尝试自动恢复打开。
8. 用户可以点击索引项打开文档。

### 9.2 打开文档流程
1. 用户在左侧索引区查看 Markdown 列表。
2. 用户点击某个 Markdown 文档。
3. 系统读取文档内容。
4. 中央内容区展示文档内容。
5. 状态栏更新当前文档状态。
6. 若打开失败，内容区展示失败提示并允许继续切换其他文档。

### 9.3 编辑与保存流程
1. 用户打开一个 Markdown 文档。
2. 用户在当前模式下编辑内容。
3. 页面进入未保存状态。
4. 用户点击保存或使用快捷键保存。
5. 系统完成保存后，未保存状态消失。
6. 若保存失败，界面保留未保存状态并展示错误原因。

### 9.4 模式切换流程
1. 用户打开一个 Markdown 文档。
2. 用户点击模式切换入口。
3. 系统切换到目标模式。
4. 当前文档内容、保存状态和上下文保持不变。
5. 用户继续编辑或阅读。

### 9.5 切换文档流程
1. 用户正在编辑或阅读当前文档。
2. 用户点击索引区中的另一篇文档。
3. 若当前文档未保存，系统弹出提示。
4. 用户选择保存后切换、放弃修改后切换，或取消切换。
5. 系统根据选择执行对应操作。
6. 切换成功后，内容区和状态栏同步更新。

### 9.6 切换工作区流程
1. 用户点击“选择工作区”或“切换工作区”。
2. 若当前文档未保存，系统弹出提示。
3. 用户确认后重新选择文件夹。
4. 系统重新建立索引并刷新页面内容。
5. 切换完成后，旧工作区相关索引和文档状态被替换为新工作区上下文。

### 9.7 点击文档内相对链接流程
1. 用户在渲染区点击相对链接。
2. 系统判断链接目标是否位于工作区内。
3. 若目标是 Markdown 文档，则在应用内打开对应文档。
4. 若目标是图片或其他资源，则按资源查看或系统默认方式处理。
5. 若目标不存在，则给出错误提示。

## 10. 交互细节要求

### 10.1 索引交互
- 单击索引项打开文档。
- 当前文档在索引中应有高亮状态。
- 索引项名称过长时应进行合理截断或完整展示处理。
- 多层目录下应保持清晰层级结构。
- 索引刷新时，应尽量避免滚动位置突然跳变。

### 10.2 模式交互
- 模式切换入口应足够明确。
- 切换反馈应直观，不应让用户误判当前模式。
- 切换过程中不应造成明显闪烁。
- 切换后若渲染尚未完全更新，应有短暂但不打断的状态反馈。

### 10.3 渲染交互
- 编辑内容变化后，预览应及时更新。
- 相对路径图片应可在工作区中正确展示。
- 相对路径链接应可在工作区中正确跳转。
- 行内公式和块级公式应具备稳定显示效果。
- 对无法渲染的公式或资源，应保留可感知的错误占位。

### 10.4 超长文档交互
- 超长 Markdown 文档打开时，界面不应长时间无响应。
- 超长 Markdown 文档滚动时应保持平滑。
- 超长 Markdown 文档在双模式切换时不应明显卡顿。
- 超长 Markdown 文档在索引切换后应尽快完成可视区域显示。
- 当性能压力过高时，允许先展示首屏和基础结构，再逐步补全后续渲染内容。

### 10.5 确认对话框交互
对于未保存文档，切换文档、切换工作区、关闭窗口时的确认提示应满足：
- 明确说明当前文档存在未保存修改。
- 至少提供“保存并继续”“不保存并继续”“取消”三类选择。
- 取消后应保持当前页面和编辑状态不变。
- 保存失败后不应继续执行原切换动作，除非用户再次明确选择放弃保存。

### 10.6 快捷键要求
MVP 建议支持以下快捷键：
- `Cmd/Ctrl + O`：选择或切换工作区。
- `Cmd/Ctrl + N`：新建文档。
- `Cmd/Ctrl + S`：保存当前文档。
- `Cmd/Ctrl + Shift + S`：另存为。
- `Cmd/Ctrl + Z`：撤销。
- `Cmd/Ctrl + Shift + Z` 或 `Cmd/Ctrl + Y`：重做。
- `Cmd/Ctrl + 1` 或自定义快捷键：切换到无感渲染模式。
- `Cmd/Ctrl + 2` 或自定义快捷键：切换到双栏编辑模式。

## 11. 页面设计约束
- 页面布局应清晰，优先服务高频编辑和阅读流程。
- 不应把主要功能隐藏过深。
- 工作区索引区、内容区、状态区职责应明确。
- 双栏模式下不应因为动态内容导致整体布局抖动。
- 页面需要兼顾浅色主题和深色主题下的可读性。
- 文本、表格、代码块、公式在小窗口尺寸下也应保持基本可读。
- 任何状态下都不应让用户无法找到返回主流程的操作入口。

## 12. MVP 页面验收点
- 空工作区状态清晰，用户知道下一步是选择工作区。
- 工作区加载状态明确，且不会阻塞界面。
- 索引区可以展示目录层级并支持点击打开文档。
- 中央内容区支持两种模式，且切换后状态不丢失。
- 未保存状态在界面中明确可见。
- 切换文档、切换工作区、关闭窗口时的未保存确认逻辑完整。
- 长文档和超长文档场景下界面仍可持续交互。
- 文件读取失败、保存失败、索引失败、公式失败、图片失败等异常均有可见反馈。

## 13. 本文档输出边界
- 本文档只说明页面结构、页面状态和交互行为。
- 本文档不展开具体技术实现方案。
- 本文档不定义底层索引结构、数据格式和具体渲染库选型。
- 具体命令接口、数据结构和性能实现细节以技术设计文档为准。
