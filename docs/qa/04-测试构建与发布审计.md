# 测试、构建与发布审计

## 1. 审计结论

核心测试、构建、版本和仓库规范命令当前全部通过，但 CI/Release 在供应链最小权限、门禁顺序、覆盖率、压力测试副作用和可重复构建方面仍有明显缺口。

发现：
- 高：2 项
- 中：7 项
- 低：3 项

审计基于提交 `b591ee34dc5e51c62a7be450df4ff44371e8a4f8` 附近的当前工作树；本次创建的 `docs/qa/` 为预期未跟踪审计产物。

## 2. 高严重度问题

### CI-H1：Release 产物可在 Rust 门禁失败前创建并上传

位置：`.github/workflows/release.yml:52-54,112-114,179-203,224-231`。

`publish` 只依赖 lint，rust-check 与其并行；tauri-action 可先创建草稿 Release 和上传二进制，最终 finalize 才等待 rust-check。

影响：失败构建仍可能留下可下载或被人工误发布的草稿资产。

建议：publish 同时 needs lint 和 rust-check；或把构建与上传拆开，验证成功后才上传。

### CI-H2：第三方 GitHub Actions 未固定到不可变 SHA

位置：
- `.github/workflows/ci.yml:21,24,29,52,57`
- `.github/workflows/release.yml:24,27,32,60,63,68,90,95,136,139,144,167,172,180,207`

使用 checkout@v4、setup-node@v4、rust-toolchain@stable、rust-cache@v2、tauri-action@v1.0.0 等可移动 tag。

影响：上游 tag 移动或供应链入侵可在持有 contents:write 和签名密钥的 workflow 中运行恶意代码。

建议：全部固定 40 位 commit SHA，注释保留版本；优先处理 release workflow。

## 3. 中严重度问题

### CI-M1：release 顶层 contents:write 过宽
位置：`.github/workflows/release.yml:7-8`。lint、rust-check 和平台矩阵均继承写权限。建议顶层 read，只给 publish/finalize job write。

### CI-M2：普通 CI 不执行 repo hygiene 和 pressure test
位置：`.github/workflows/ci.yml:61-77`。两项只在 release 中运行。建议 repo:check 加入所有 PR；pressure 可分轻量 PR 版和完整 main/tag 版。

### CI-M3：前端无覆盖率 provider 与阈值
位置：`vitest.config.ts:5-9`、`package.json:11,26-36`。`vitest run --coverage` 因缺少 `@vitest/coverage-v8` 失败。建议增加 provider、test:coverage 和逐步提升阈值。

### CI-M4：pressure wrapper 会修改仓库目录
位置：`scripts/pressure-test-workspace.mjs:67-70,263-282`。创建根目录 `.pressure-tmp`，并覆盖 `handoff/pressure-test-report.json`；中止时可留下大量文件。

建议：使用 os.tmpdir/mkdtemp；报告默认 stdout 或 CI artifact；`.pressure-tmp/` 加入 ignore 兜底。

### CI-M5：压力测试输出评级与失败阈值不一致
位置：`src-tauri/tests/pressure_bench.rs:164-174,190-195`、`pressure-test-workspace.mjs:239-250`。超过 5 秒标记 slow，但只有超过 30 秒才失败，20 秒严重回归仍为绿。

建议：quality=slow 应非零退出，或定义 CI SLA/历史倍率门槛。

### CI-M6：Cargo CI 命令未显式使用 --locked
位置：`.github/workflows/ci.yml:74,77`、`release.yml:103,109`、pressure script。建议 test/clippy 都加 `--locked`。

### CI-M7：版本策略文档与实际门禁矛盾
位置：`VERSIONING.md:18`、`scripts/check-version-sync.mjs:59-76`。参见文档一致性报告 DOC-M4。

## 4. 低严重度问题

### CI-L1：生产包超过 Vite 500 kB 默认警戒线
构建主 JS 553.26 kB，gzip 169.12 kB。建议动态导入、manualChunks 或基于测量调整阈值。

### CI-L2：依赖维护存在积压和弃用类型包
`package.json:29` 的 `@types/dompurify` 已弃用（DOMPurify 自带类型）；另有多个 patch/minor/major 可升级。建议先移除弃用包，再分批升级并回归。

### CI-L3：repo hygiene 硬编码历史版本路径
位置：`scripts/check-repo-hygiene.mjs:44-46,149-151`。下一版本需手工改脚本，规则也无法普遍识别状态漂移。建议从当前版本和 releases 目录动态派生。

### CI-L4：本地 updater 签名私钥文件权限过宽

位置：`/Users/joker/Code/RuT0MarkFlow/.tauri/signing-key.key`

证据：实际权限为 `-rw-r--r--`（0644），同机其他用户可读。该文件已被 `.gitignore` 忽略且未被 Git 跟踪，但"未提交到 Git"不等于私钥本地安全。

影响：多用户机器、共享 runner、备份或诊断采集环境中可能泄露发布私钥。

建议：私钥权限设为 `0600`；核对历史备份、日志和 CI 环境是否曾复制该密钥；文档中不应把 gitignored 等同于私钥安全。

### CI-L5：updater 公钥一致性已复核

本次独立复核结果：`tauri.conf.json` 中 `pubkey` 字段 base64 解码后 SHA-256 为 `a336f679f6cb4ee7a50a8b6cde5633d2999c01a2eb8377d6b7e7b20be04c6c0c`；本地 `.tauri/signing-key.key.pub` base64 解码后 SHA-256 相同。两者一致。

仍需外部环境验证的事项：GitHub Secret 是否使用该本地私钥；远端 `latest.json` 和 `.sig` 是否由该密钥生成；真实应用内升级验签链路。
