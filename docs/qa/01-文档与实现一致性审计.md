# 文档与实现一致性审计

## 1. 审计结论

当前代码与版本配置实际为 `v0.1.0`（融合基线版本），核心功能和自动化验证总体可运行，但正式文档存在明显状态漂移和接口过期。

发现：
- 高：2 项
- 中：7 项
- 低：4 项

## 2. 高严重度问题

### DOC-H1：当前版本状态在正式文档之间互相矛盾

证据：
- `docs/README.md:10,56` 将 `v0.1.0` 描述为正在实施。
- `docs/05-验收记录.md:116-127` 将 `v0.1.0` 标记为 `in_progress`。
- `docs/releases/README.md:11-16`、`docs/releases/v0.1.0/规划需求.md:5`、`docs/releases/v0.1.0/更新日志.md:3-15` 则声明 `v0.1.0` 已完成验收。
- `package.json:3`、`version-manifest.json:1-4`、`src-tauri/Cargo.toml:2`、`src-tauri/tauri.conf.json:3`、`src/version.ts:8` 均为 `0.1.0`。
- 本次 `pnpm version:check -- --tag v0.1.0` 通过。

影响：文档索引和验收事实入口不可信，维护者或自动化代理可能重复实施已完成工作，或错误阻止后续版本启动。

建议：已统一为 `v0.1.0` 融合基线，不再保留中间版本；更新根文档、验收记录及仍引用旧状态的需求/技术/任务文档。

### DOC-H2：发布门禁描述与工作流实际行为不一致

证据：
- `docs/releases/v0.1.0/更新日志.md` 与 `docs/04-开发任务清单.md:1186-1205` 将压力测试等描述为发布前阻断门禁。
- `.github/workflows/release.yml:52-54,112-115` 中 `publish` 仅依赖 `lint`，与 `rust-check` 并行。
- `.github/workflows/release.yml:179-222` 可在 Rust 检查结束前构建并上传草稿 Release 资产。
- `.github/workflows/release.yml:224-245` 仅在将 draft 转正式 Release 时等待 `rust-check`。

影响：文档容易让维护者误以为失败时不会生成或上传任何二进制，实际只是不会最终发布。

建议：若要求严格门禁，让构建发布 job 同时依赖 lint 与 rust-check；否则文档明确表述为“正式发布门禁，不阻止草稿资产构建”。

## 3. 中严重度问题

### DOC-M1：无感渲染模式被描述为轻编辑，实际为只读预览

证据：
- `docs/01-需求文档.md:99-100`、`docs/02-页面与交互说明.md:90-96,216-223` 描述阅读与轻编辑/当前模式编辑。
- `src/components/content/ContentArea.tsx:261-281` 沉浸模式只渲染 HTML。
- 编辑入口只在 `ContentArea.tsx:286-302` 的双栏分支。

影响：产品能力承诺高于实现，MVP 验收口径不准确。

建议：现阶段改称“沉浸式只读预览”；块级轻编辑已实现（无感编辑模式）。

### DOC-M2：新建未落盘文档没有真正的双模式视图

证据：
- `docs/02-页面与交互说明.md:158-163` 要求新建文档可切换模式。
- `ContentArea.tsx:232-247` 在判断 viewMode 前直接返回单 textarea。
- `Toolbar.tsx:56` 模式按钮仍可用。

影响：模式状态变化但界面不变化，形成假成功交互。

建议：实现新建文档双模式，或在未落盘状态禁用模式切换并同步文档约束。

### DOC-M3：技术设计中的 Tauri 命令接口表已过期

证据：
- `docs/03-技术设计文档.md:347-364` 列出不存在的 `get_workspace_status`、`create_document`，且 `save_document_as` 参数过期。
- 实际注册列表见 `src-tauri/src/lib.rs:33-50`，包含文档未列出的 watcher、保存路径选择、dirty state、updater 命令。
- 实际 `save_document_as` 见 `src-tauri/src/commands/document.rs:143-147`。
- 新建实际是前端临时状态：`src/app/App.tsx:536-552`。

影响：按技术文档开发或联调会直接调用失败。

建议：按实际注册命令、参数和职责重写命令表。

### DOC-M4：版本策略文档与版本检查脚本冲突

证据：
- `VERSIONING.md:5,18` 允许 frontend/backend 独立演进，且漏列 `src/version.ts`。
- `scripts/check-version-sync.mjs:59-81` 强制 package、Cargo、Tauri、manifest、frontend/backend、fallback 版本完全一致，并校验 workspace schema fallback。

影响：维护者按文档操作会被门禁拒绝。

建议：明确选择统一版本或组件独立版本，并同步修改文档和脚本。

### DOC-M5：验收记录的源码行号大量失效

示例：
- `docs/05-验收记录.md:13` 指向旧 `App.tsx:454`，当前选择工作区逻辑在 `App.tsx:595-627`。
- `docs/05-验收记录.md:16` 的打开文档行号已漂移到 `App.tsx:633-677`。
- watcher、ContentArea 和 viewMode 等多处证据同样失效。

影响：验收记录不能继续作为可追踪证据。

建议：刷新证据；除行号外同时引用符号名、测试名、commit SHA 和命令输出。

### DOC-M6：测试计数已过期

证据：
- `docs/releases/v0.1.0/更新日志.md` 记录 193 tests/26 files。
- 本次实际为 245 tests/32 files；Rust 为 97 单元测试 + 1 pressure integration test。

影响：无法区分历史验收结果和当前回归规模。

建议：测试记录增加日期、commit 和“验收当时/当前复核”标签。

### DOC-M7：普通 CI 未执行文档所称的 repo hygiene 与 pressure 门禁

证据：
- `.github/workflows/ci.yml:64-77` 只运行版本、前端测试/build、Rust test/clippy。
- `repo:check` 与 `pressure:test` 只在 release workflow 中执行。

影响：PR 阶段不能提前发现规范和压力回归。

建议：至少将 repo:check 加入 PR；压力测试可采用 PR 轻量版、main/tag 完整版。

## 4. 低严重度问题

### DOC-L1：README 对本地构建签名变量的要求表述过度
`README.md:43-58` 容易让开发者误以为没有 updater 私钥就不能进行普通本地 Tauri 构建。建议区分开发构建和正式签名发布构建。

### DOC-L2：更新日志章节标题规范
`docs/releases/v0.1.0/更新日志.md` 与文件前部 complete 状态冲突。应改为“关键变更”。

### DOC-L3：规划文档引用已清理的临时 HANDOFF
`docs/releases/v0.1.0/规划需求.md` 引用不存在的任务交接文件。应注明临时工件已按流程清理，并改引永久记录。

### DOC-L4：repo hygiene 能力被文档夸大
`scripts/check-repo-hygiene.mjs:77-89,131-151` 只检查有限版本路径和模式，未发现 DOC-H1。文档不应把通过该脚本等同于整套文档状态一致。

## 5. 已验证一致项

- 版本元数据均为 `0.1.0`，版本检查通过。
- v0.1.0 窗口关闭保护已实现。
- 设置页和更新器迁移已实现。
- updater 配置、权限与插件注册存在。
- Release 平台矩阵和 bundle 目标基本符合 v0.1.0 规划。
- Windows BOM/GBK/GB18030 与 CRLF 兼容逻辑存在。
- 扫描后台化和 watcher 已实现。
