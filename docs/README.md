# 文档索引

## 1. 文档目标
- 作为 `RuT0MarkFlow` 文档集的统一入口。
- 帮助区分 `v0.1.0` 已完成基线与 `v0.1.x` 后续优化计划。
- 明确不同文档在产品、设计、实施、验收中的职责边界。

## 2. 当前基线
- `v0.1.0` 已于 2026-07-06 完成 MVP 验收，证据见 `docs/05-验收记录.md`。
- `docs/` 根目录下的 `01` 到 `05` 为 MVP 基线文档。
- `docs/releases/` 承接 `v0.1.0` 之后的后续版本文档；`v0.1.1` 稳定化、`v0.1.2` Windows 跨平台修复、`v0.1.3` GitHub 自动更新均已完成验收，`v0.1.4` BUG 修复与设置界面正在实施中，`v0.2.0` 写作体验增强已完成规划尚未实施。
- 后续版本主线与节奏由 `docs/releases/路线规划.md` 统一约束。
- 版本号规则以仓库根目录 `VERSIONING.md` 为准。

## 3. 分阶段阅读顺序

### 阶段 A：MVP 产品边界
- `docs/01-需求文档.md`

用途：
- 确认 `v0.1.0` 做什么、不做什么。
- 作为当前实现与验收的产品基线。

### 阶段 B：MVP 页面交互与技术设计
- `docs/02-页面与交互说明.md`
- `docs/03-技术设计文档.md`

用途：
- 确认界面结构、交互状态和用户可见反馈。
- 确认模块职责、数据结构、性能和可扩展性边界。

### 阶段 C：MVP 执行任务与验收证据
- `docs/04-开发任务清单.md`
- `docs/05-验收记录.md`

用途：
- 跟踪历史 MVP 实施链路和后续版本任务拆解。
- 对照需求文档保留验收证据和版本状态。

### 阶段 D：后续版本规划与变更记录
- `docs/releases/README.md`
- `docs/releases/路线规划.md`
- `docs/releases/v0.1.1/规划需求.md`

- `docs/releases/v0.1.3/规划需求.md`
- `docs/releases/v0.1.3/更新日志.md`
- `docs/releases/v0.1.4/规划需求.md`
- `docs/releases/v0.1.4/更新日志.md`
- `docs/releases/v0.2.0/规划需求.md`

用途：
- `路线规划.md` 定义 `v0.1.1` 之后的功能演进主线与各版本范围概要。

## 4. 更新规则
- MVP 产品范围变化，优先更新 `docs/01-需求文档.md`。
- 页面结构、状态、交互变化，更新 `docs/02-页面与交互说明.md`。
- 模块职责、数据结构、命令接口、性能策略变化，更新 `docs/03-技术设计文档.md`。
- 开发拆解、优先级、发布任务变化，更新 `docs/04-开发任务清单.md`。
- 验收结果、回归结论、版本完成状态变化，更新 `docs/05-验收记录.md`。
- 后续版本计划、版本内变更和发布收敛结果，更新 `docs/releases/` 对应版本目录。

## 5. 推荐维护节奏
- 每个新版本开始前，先在 `docs/releases/` 下创建或更新对应版本的“规划需求”与“更新日志”。
- 每个版本收尾时，至少同步 `docs/05-验收记录.md`、必要的 `docs/03-技术设计文档.md` 和根目录 `VERSIONING.md`。
- 任何已经进入发布分支的改动，都不应只改代码而不更新对应文档。
